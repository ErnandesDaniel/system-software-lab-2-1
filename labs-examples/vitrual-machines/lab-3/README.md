# lab-3: Mark-and-Sweep Garbage Collector (NASM)

Реализация алгоритма «Полная сборка методом «пометить и освободить» (Mark and Sweep)
для NASM-таргета — как **обычная MyLang-программа**, а не встроенный рантайм.

## Что делает программа

Два сценария в одном файле, разделённые флагом `gc_enabled`:

### Сценарий 1: GC ON

Цикл аллоцирует объекты по ~1 KB и запускает GC каждые 10 итераций.
GC находит недостижимые объекты (на которые нет ссылок) и освобождает их.
Память остаётся стабильной — программа работает без роста потребления.

### Сценарий 2: GC OFF

Тот же цикл, но GC отключён. Память растёт, пока `malloc` не вернёт 0 (OOM).
Программа детектирует это и выводит статистику.

### Сравнение

| | GC ON | GC OFF |
|---|---|---|
| Потребление памяти | ~10-30 KB стабильно | Растёт до OOM |
| Макс. итераций | Неограниченно | ~200-400 (зависит от RAM) |
| Итог | Работает вечно | Падает с OOM |

## Суть

GC в этой лабе — не встроенный в компилятор, а **обычная функция на MyLang**.
Он «защищает» ручные вызовы `malloc`/`free` от утечек: вместо того чтобы `free`
вручную, программа вызывает `gc_run()`, которая сама находит и освобождает
недостижимые объекты.

Это **та же логика, что в Java/Go/C#**, но без автоматического триггера.
Разница только в том, кто решает *когда* вызывать GC — рантайм или ты.
